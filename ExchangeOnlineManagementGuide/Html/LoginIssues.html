Common Issues with logging in to OWA and Exchange

Written by Charlie King - Microsoft Consulting - May 2016
Rule #1

Users are liars

Resolution B

If you cannot find their mailbox, and you have tried searching by UserID, displayName, and FirstName/LastName, then they probably do not have a mailbox, and you should inform them of that and direct them toward the ServiceNow request for a new mailbox.

Resolution C

If they claim to have already requested a mailbox, ask them corresponding to the RITM number for the task and confirm that it was completed with the Access Management team.


Case

A user can log in to OWA, but they can't log in to Outlook 2013.

Resolution A

Check their account in the O365 Admin Center to see if they have the correct license type. If they have an "Exchange Online Kiosk" license, then they will not be able to log into Outlook 2013, the Kiosk license is for OWA only. You can change their license type and it will fix their issue almost immediately.

Resolution B

In extremely rare cases, there may be an error with the Azure Active Directory User Account. The account will need to be deleted from Azure Active directory.

Note: Please proceed with extreme caution when attempting the following steps.

The scripts to remove the account are as follow:

First, remove the Azure AD user

Remove-MsolUser -UserPrincipalName <abc1234@dep.ca.gov>

Next, hard-delete the user

Remove-MsolUser -UserPrincipalName <acb1234@dep.ca.gov> -RemoveFromRecycleBin

Now it will take at least one more sync in order for the user account to be re-created, maximum 30 minutes must elapse. During this time they should have been totally removed and you wouldn't be able to search for them.

Once the user account appears in O365 again, get the GUids for both mailboxes. You need both of them to merge the mailboxes back after they come back.

First, the new mailbox that was automatically created when the user object synced back into Azure Active Directory:

Get-Mailbox -Identity <abc1234@dep.ca.gov> | fl "guid"

Next, the old mailbox that is currently in a soft-deleted state. It's the data from this mailbox that you will be merging into the new mailbox.

Get-Mailbox -Identity <abc1234@dep.ca.gov> -SoftDeletedMailbox | fl "guid"

Now kick off a mailbox merge for the two mailboxes

New-MailboxRestoreRequest -SourceMailbox <OldGUID> -TargetMailbox <NewGUID>

The merge request will begin and it will go in the background. At some point you will be alerted that the mailbox restore completed.

You will need to reset the user's license, possibly restore their archive with a mailbox restore on the archive mailboxes. If the user was on Litigation Hold before they were deleted, they need to be placed on Litigation Hold again.
